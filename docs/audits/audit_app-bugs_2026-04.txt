SECTION 1 — PROJECT RULE VIOLATIONS: invalidateQueries
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The WORKFLOW.md hard rule is: use refetchQueries, NEVER invalidateQueries.
Reason: offline-first caching — invalidateQueries marks data stale but
does not re-fetch, so components can render empty/stale after a write.

Every call below needs to change from:
  queryClient.invalidateQueries({ queryKey: [...] })
to:
  queryClient.refetchQueries({ queryKey: [...] })

SECTION 2 — MISSING MODAL SAFETY GUARDS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WORKFLOW.md rule: any modal with user input must have both:
  onInteractOutside={(e) => e.preventDefault()}
  onEscapeKeyDown={(e) => e.preventDefault()}
on the DialogContent. Missing these lets users accidentally dismiss
a modal mid-save, corrupting in-progress form state.

─────────────────────────────────────────────────────────────
BUG-018 | HIGH | pages/EditRoute.jsx | Lines 336 and 374
What : Two DialogContent elements (rename route, delete confirmation)
       have no onInteractOutside or onEscapeKeyDown guards
Why  : User can tap outside or press Escape mid-rename, losing the edit;
       delete confirmation can be dismissed by accident
Fix  : Add both guards to both DialogContent elements
─────────────────────────────────────────────────────────────
BUG-019 | HIGH | pages/RouteEditor.jsx | Line 375
What : The address-move DialogContent has no interaction guards
Why  : Tapping outside during an address reassignment dismisses the
       dialog while the mutation may still be in flight
Fix  : Add onInteractOutside and onEscapeKeyDown prevention
─────────────────────────────────────────────────────────────
BUG-020 | MEDIUM | pages/VacationRequests.jsx | Line 256
What : The vacation request approval dialog has no interaction guards
Why  : Boss can accidentally dismiss mid-approval by tapping the backdrop
Fix  : Add both guards to the DialogContent

SECTION 4 — localStorage WITHOUT try/catch
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WORKFLOW.md rule: localStorage access must be wrapped in try/catch.
On some devices (private browsing, storage quota exceeded, iOS Safari
restrictions) localStorage throws synchronously and crashes the whole page.

─────────────────────────────────────────────────────────────
BUG-022 | HIGH | pages/BulkScanOptimize.jsx | Lines 16 and 29
What : saveBulkSession() calls localStorage.setItem without try/catch.
       clearSavedBulkSession() calls localStorage.removeItem without try/catch.
       (loadSavedBulkSession at line 23 IS correctly wrapped — inconsistent)
Why  : On a worker's phone in private mode or with full storage, saving
       the bulk scan session crashes the entire page, losing all scan work
Fix  : Wrap both localStorage.setItem and localStorage.removeItem calls
       in try/catch the same way loadSavedBulkSession is already written

SECTION 5 — GEOCODING / API KEY SECURITY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

─────────────────────────────────────────────────────────────
BUG-023 | MEDIUM | components/route/RouteOptimizeModal.jsx | Line ~190
What : The reverse-geocode URL is built with the full MapQuest API key
       in the query string and fetched client-side. The full URL (including
       the key) appears in the browser's Network tab and console errors.
       The 401 error in your own logs exposed the key: 7fGk1x8MnojBKZ1syLiFv4VNFZ3B6B45r
Why  : Anyone using DevTools on a shared device can copy the API key.
       That key can then be used to make API calls billed to your account.
Fix  : Move the reverse-geocode call to a Base44 serverless function so
       the key never appears client-side; or at minimum rotate the key
       since it has already been exposed in session logs.
─────────────────────────────────────────────────────────────
BUG-024 | LOW | components/route/RouteOptimizeModal.jsx | Line ~192
What : The reverse-geocode call has no 401/error handling branch —
       when it fails (as it does in current logs) the catch block
       silently swallows it. No fallback message is shown.
Why  : Worker sees "locating..." indefinitely instead of their actual
       GPS coordinates if the reverse geocode key is invalid
Fix  : On non-ok response, fall back to showing raw coordinates
       (lat.toFixed(5), lng.toFixed(5)) instead of the readable address

BUG-025 | HIGH | pages/AddressPool.jsx | Lines ~86 and ~121
What : Bulk-delete loop calls deleteAddressMutation.mutate(id) for each
       selected address inside a forEach with no disabled guard on the
       "Delete Selected" button after the first click
Why  : Double-tapping the delete button fires two full delete loops,
       attempting to delete already-deleted records and causing errors
Fix  : Set a deleting state flag to true before the loop and disable
       the button; reset in a finally block
─────────────────────────────────────────────────────────────
BUG-026 | MEDIUM | pages/ScanRouteSetup.jsx | Line ~173
What : Route is written to queryClient cache (setQueryData) immediately
       after Route.create(), but before the Address.create() loop runs.
       If any Address.create() call fails, the cache shows a route
       with the wrong address count.
Why  : WorkerRouteDetail loads the cached route and shows incorrect totals;
       the worker thinks all addresses are there when some are missing
Fix  : Only write the route to cache after the full address loop completes
       (or re-fetch the route from the server after all creates)
─────────────────────────────────────────────────────────────
BUG-027 | MEDIUM | components/route/MoveAddressesModal.jsx | Lines ~59–104
What : handleMoveToExisting loops through selected addresses and calls
       Address.update() sequentially with no error collection. If one
       update fails mid-loop the source and target route address counts
       are now inconsistent and there is no recovery path.
Why  : Worker moves 8 addresses, 3 succeed, 5 fail — now both routes
       show wrong counts and there is no way to know which addresses moved
Fix  : Collect all failed address IDs and surface them to the user;
       consider rolling back by re-updating the successfully moved ones

BUG-028 | LOW | pages/WorkerSettings.jsx | Line ~41
What : When UserSettings.create() is called, the `...data` spread
       includes whatever the caller passes — no validation that required

What : Mark-read, mark-all-read, and delete handlers all call
       invalidateQueries (already noted above) but also have no disabled
       guard — rapid tapping can fire multiple mark-read mutations
Why  : Duplicate mutation calls; notifications may flicker or double-update
Fix  : Add a processing state flag and disable action buttons during mutation
─────────────────────────────────────────────────────────────
BUG-030 | LOW | components/services/OptimizationService.jsx | Lines 204 and 243
What : geocodeWithHere and geocodeWithMapQuest log the full API key in
       error messages via console.error (the key appears in the URL string)
Why  : Any error logged during geocoding exposes the full key in the
       console, visible to anyone with DevTools open on the device
Fix  : Redact the key from logged URLs:
       url.replace(/key=[^&]+/, 'key=REDACTED') before logging
